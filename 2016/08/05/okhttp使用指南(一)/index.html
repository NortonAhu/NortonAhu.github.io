OkHttp 使用指南(一)–Calls

手打翻译原文，若转载本文请注明出处

Calls(调用)

 Http 客户端的作用就是接受你的发送请求并且获得响应，这个在理论上是很简单但是实际上是很复杂的。

Request(请求)
 每一个 Http 请求都是包含了一个 URL 、一个 Method(例如 POST GET) 、header 列表，Request 也会包含一个数据体：指定内容类型(Content-type) 的数据流。

Response(响应)
 响应是为了回应 Request 请求，它包含了返回码(200,404 等)，header 信息以及自身的数据体信息。

Rewriting Request(重写请求)
 当你准备使用 OkHttp 来发送一个 Http 请求，并且你将这个请求描述为最高级别：「使用当前的 headers 信息获取该 URL 的信息」。为了正确和高效率，OkHttp 会在发送请求之前重写请求（主要是 headers 信息）。
 OkHttp 会加上原来请求信息的头部信息中缺失的部分，例如 content-Length, Transfer-Encoding, User-Agent, Host, Connection, and Content-Type。并且还会上一个 Accept-Encoding 到请求的头部中，除非你已经指定该头部信息，如果你有一个 cookie 信息，也会将这个 cookie 信息加到 headers 中的。
 有些请求会有 cache 响应，当这种 cache 响应没有刷新的时候，OkHttp 可以通过一种 有条件 的 GET 请求来下载刷新这个 cache ，当然这个 cache 要比当前的要新。这就需要在 header 中增加 If-Modified-Since 或者 If-None-Match 。

ReWriting Response(重写响应)
 如果采用了透明的压缩，OkHttp 会删除相应的响应 headers 中的 Content-Encoding 和 Content-Length，因为这些将不会应用到响应信息的解压缩。
 如果有条件的 GET 请求成功了，来自网络的响应和缓存会按照规范来定向合并。

Follow-up Requests(持续请求)
 当你请求的 Url 已经被移动了，WebServer 会返回一个响应码类似于 302 这种来告知去请求一个新的 Url。OkHttp 会根据这个新的 Url 来查找最终的响应。
 如果响应信息是一个授权质疑。OkHttp 会查询 Authenticator来满足这种质疑(前提是已经有了一个配置信息)。如果认证器能够提供一个授权证书，那么 OkHttp 将会加上这个证书再一次请求。

Retrying Request(重新请求)
 有时候连接会失败原因可能是连接池已经满了和断开连接，也有可能是 WebServer 自己是不可达的。如果有可选的路由的话 OkHttp 重现选择一条来请求。

Calls
 通过重写，重定向、持续请求和重新请求，你简单的请求会导致大量的请求和响应。OkHttp 使用 Call 满足你的请求任务的模型来通过无论多少中间必要的请求和响应。通常这不是很多!但是如果你请求的 url 被重定向了或者由于故障转移更换了 IP 地址这对你来说便捷的是请求代码会持续工作。

 Calls 将会通过下面两种方式来进行请求：

  • Synchronous：在响应未到达之前你的进程会阻塞；
  • Asynchronous：你的请求会在任一线程中排队，通过 call back 来在另外一线程中执行。

 Calls 可能会在任意线程中被取消。如果请求没有完成的话会导致调用失败。当调用被取消的时候代码中执行读取操作或者写入操作可能会抛出 IOException

Dispatch(调度)
 对于同步调用，在你的线程中对于同时请求会有很多响应需要处理。太多相同的连接会浪费掉你的资源。但很少潜在的伤害。
 对于异步请求，Dispatcher 使用最大请求次数的方案，你可以设置最大的 per-webserver （默认是 5 个）和 overall（默认64个连接）。
