OkHttp 使用指南(二)–Connections


手打翻译原文，若转载本文请注明出处
 尽管你使用 OkHttp 可能只需要提供的是一个 Url，但是在连接服务器的时候他用到了三个组件：URL、ADDRESS、ROUTE。

URLS
 URLs (类似于 https://github.com/square/okhttp) 是对 HTTPS 和 Internet 访问的基本样式，除了是网络上普遍、分散命名访问网络外，他还可以指定怎么访问网络资源。

URLs 是抽象的：

  • 他们指定的请求可能是纯文本的 (http://) 或者是加密的 (https://) 但不指定那一个加密算法被使用。也不指定怎么对host的认证(HostnameVerifier) 也不能指定哪一个授权证书被信任(SSLSocketFactory)
  • 也不会指定那个代理服务器被使用以及怎么验证代理服务器。

他们也是具体的：每一个url都会有一个指定路径(/square/okhttp)和查询(?q=sharks&lang=en)。每一个 host 都会有很多 urls。

Address
 address 指定与连服务端和服务器所有的静态配置信息（例如 github.com）:端口号、HTTPS 设置、首选的网络协议(SPDY 还是 HTTP/2)。
 URLs 共享地址也许潜在的共享 TCP 端口，共享连接有一个很明显的性能优势：低延迟、高吞吐、节约电源。OkHttp 使用
ConnectionPool 来实现 HTTP/1.x 和多个 HTTP/2 、SPDY 连接的复用。
 在 OkHttp 中大部分地址的属性是来自 URL (端口、模式、主机名)，其他都是来自 OkHttpClient 本身。

Routes（路由）
 路由在实际的连接中支持必要的动态信息，这是一个根据 DNS 查询获得特定的 IP 地址、代理服务器的选择 (ProxySelector) 和确定 TLS 的版本。
 也许一个 IP 地址对应的是多个路由选择，例如 一个服务器被多个数据中心管理 这样在 DNS 响应中会获得多个 IP 地址。

Connections
 当你通过 OkHttp 来访问一个连接的时候，下面是其主要流程：

  • 通过 URL 和 OkHttpClient 的配置信息创建出一个 address，这个 address 指定了我们该怎么来访问服务器；
  • 使用这个 address OkHttp 尝试从 Connection Pool 找到一个 address 连接；
  • 如果没有在 Connnection Pool 找到一个连接话，尝试选择一个 Route，这意味需要使用进行一个 DNS 请求来获得一个 服务器 IP 地址，然后选择 TLS 版本，如果有必要的话会选择代理服务器；
  • 如果是一个新的路线，这个连接或者是直接的 socket 连接一个 TLS 通道（ HTTP 或者 HTTPS），或者直接一个 TLS 连接。这个会进行必要的 TLS 握手操作。
  • 最后会发送请求和接受响应。

 如果在连接的过程中遇到了错误，他会选择另外一个路线进行重试访问。这在服务器的其中一个 IP 不可达的状态下 OkHttp 可以恢复过来。这对于连接池满了或者尝试使用的 TLS 版本不支持的情况。
 一旦请求得到了响应，这个连接就会回到连接池中所以这个在未来的连接中可以被重复使用。连接池也会定期清理那些长时间不使用的连接。
